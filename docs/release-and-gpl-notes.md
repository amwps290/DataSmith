# 发布说明与 GPL 注意事项

本文档用于 Kudu 对外发布前的合规自查，不构成法律意见。正式发布前，建议至少核对一次上游仓库的许可证原文、版权声明和品牌使用限制。

## 当前结论

- Kudu 基于上游项目 [Rabb1tQ/DataSmith](https://github.com/Rabb1tQ/DataSmith) 继续开发
- 你可以更换名称、图标、界面文案和产品说明
- 你不能移除或覆盖上游已有的版权和许可证要求
- 如果对外发布二进制，仍需同时提供与该版本对应的源代码

## 发布前必须确认的事项

### 1. 许可证链路完整

发布仓库中应至少包含：

- 上游对应版本的 `LICENSE` 原文
- 当前项目的 `README`
- 与上游关系说明
- 当前版本的版权或维护说明

### 2. 品牌替换完成

建议确认以下信息已经统一：

- 软件名称
- 应用标题
- 仓库名称
- 安装包元数据
- 图标与品牌标识

### 3. 源码分发方式明确

如果你发布安装包、可执行文件或其他二进制产物，需要确保用户可以获取对应版本的完整源码。

## 可以修改的内容

- 软件名称
- 包名与产品名
- 图标与视觉标识
- 界面文案
- README 与发布页内容

## 不应擅自移除的内容

- 上游版权声明
- 上游许可证文本
- 源码中已有的许可证头
- 可能属于上游项目的品牌或商标信息

## 发布页建议包含的信息

- 项目简介
- 支持的数据库类型
- 核心能力说明
- 与上游项目的关系
- 许可证说明
- 源码仓库地址
- 二进制与源码的获取方式

## 建议的发布顺序

### 第一阶段

- 确认最终名称与图标
- 完成 README
- 补齐 `LICENSE`
- 写清楚上游来源

### 第二阶段

- 完成首个公开版本打包
- 在发布页中明确版本定位
- 说明当前功能边界和已知限制

### 第三阶段

- 补截图和安装说明
- 补变更记录
- 再扩展路线图与后续计划

## 发布前检查清单

- 确认 GPL-3.0 相关文件已完整保留
- 确认品牌替换已经完成
- 确认 `README` 与发布说明可直接对外使用
- 确认 `npm run build` 通过
- 确认 `cargo check --manifest-path src-tauri/Cargo.toml` 通过
